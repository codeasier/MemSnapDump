# MemSnapDump src-layout、测试标准化与覆盖率补全设计

## 1. 背景与目标

当前项目采用传统平铺目录结构：`base/`、`simulate/`、`tools/`、`util/`、`ut/`。这类结构在仓库根目录直接运行时问题不大，但存在几个明显限制：

- 不符合标准 Python `src-layout` 打包习惯，不利于 `pip install -e .`
- 测试依赖 `sys.path` 注入与仓库根目录假设，真实安装场景覆盖不足
- 测试执行方式不统一，缺少标准覆盖率统计闭环
- 核心模块虽已有一定测试，但缺少基于覆盖率和复杂度的定向补测

本次改造目标是：

1. 将项目改造成标准可安装的 `src/memsnapdump/...` 结构
2. 将测试迁移为 `tests/...` 并统一到 `pytest`
3. 建立覆盖率统计能力，并基于结果补齐关键测试
4. 在迁移过程中识别并落地少量高收益、低风险的源码优化点

## 2. 范围与非目标

### 2.1 本次范围

- 新建 `pyproject.toml`，采用标准打包入口
- 将源码迁移到 `src/memsnapdump/`
- 将 `ut/` 迁移到 `tests/`
- 接入 `pytest` 与 `pytest-cov`
- 跑通现有测试并统计覆盖率
- 针对关键低覆盖模块补充测试
- 更新 README 中安装、测试、执行方式
- 修正因 `src-layout` 暴露出的 import、路径、入口问题
- 对关键核心模块做少量结构优化或重复逻辑收敛

### 2.2 非目标

- 不做大规模架构重写
- 不做全项目 API 重新设计
- 不强制将全部存量测试重写成纯 `pytest` 风格
- 不在本轮引入复杂的新工具链（如 lint、type checker 全量治理）
- 不将本次任务扩展为 simulate 子系统的二次大重构

## 3. 目标目录结构

改造后的推荐结构如下：

```text
MemSnapDump/
├── pyproject.toml
├── README.md
├── src/
│   └── memsnapdump/
│       ├── __init__.py
│       ├── base/
│       ├── simulate/
│       ├── tools/
│       └── util/
├── tests/
│   ├── base/
│   ├── simulate/
│   ├── tools/
│   ├── util/
│   ├── test_data/
│   └── conftest.py
└── docs/
```

### 3.1 源码迁移原则

- 保留现有模块边界：`base`、`simulate`、`tools`、`util`
- 不在目录迁移这一步同时重命名大量模块
- 包内导入统一改为相对导入或 `memsnapdump.*`
- 对外公共使用路径以安装包 `memsnapdump` 为准

### 3.2 测试迁移原则

- `ut/` 迁移到 `tests/`
- `ut/test-data/` 迁移到 `tests/test_data/`
- 删除基于 `sys.path` 注入的自定义测试入口
- 测试数据访问统一改为基于文件路径解析，不依赖当前工作目录

## 4. 打包与执行方案

### 4.1 打包配置

项目使用 `pyproject.toml` 管理以下配置：

- 项目元数据
- setuptools `src` 包发现
- 测试依赖与配置
- coverage 配置

### 4.2 安装方式

迁移完成后，项目应支持标准开发安装：

```bash
pip install -e .
```

### 4.3 工具入口策略

本轮优先保证模块运行方式可用，例如：

```bash
python -m memsnapdump.tools.split ...
python -m memsnapdump.tools.dump2db ...
```

如改造成本较低，可顺手补充 console scripts，但不作为硬性目标。

## 5. 测试与覆盖率方案

### 5.1 测试框架策略

采用 `pytest` 作为统一测试入口，但迁移策略强调兼容优先：

- 先让现有 `unittest` 测试在 `pytest` 下直接运行
- 不要求一次性重写所有旧测试风格
- 新增测试优先采用更简洁的 `pytest` 写法
- 通过 `conftest.py` 收敛公共 fixture、测试路径和数据辅助逻辑

### 5.2 覆盖率策略

采用 `pytest-cov` 统计覆盖率，典型执行方式如下：

```bash
pytest --cov=memsnapdump --cov-report=term-missing
```

覆盖率用于识别高风险盲区，而非追求单一总数字。补测优先级按以下顺序排序：

1. `simulate` 核心模块的低覆盖分支
2. `src-layout` 迁移后最容易出问题的 import / 路径 / 入口逻辑
3. 工具脚本的关键路径
4. 易于补齐但能显著提高稳定性的纯逻辑函数

### 5.3 预期重点补测模块

优先关注以下模块：

- `memsnapdump.simulate.simulated_caching_allocator`
- `memsnapdump.simulate.range_ops`
- `memsnapdump.simulate.snapshot_mutator`
- `memsnapdump.simulate.replay_executor`
- 关键工具入口模块

### 5.4 预期重点补测场景

- 地址/区间判断与边界值
- segment / block 查找失败分支
- merge / split / gap 查找等关键路径
- workspace 容错路径
- 回放流程中 hook 调用和状态变更链路

## 6. 源码优化识别方法

本轮源码优化不以“大改”为目标，而按以下三条线识别：

### 6.1 迁移暴露问题

`src-layout` 会暴露出若干真实问题，需要优先修正：

- 平铺 import 假设
- 自定义 `sys.path` 注入
- 工作目录依赖
- 测试数据路径写死

### 6.2 复杂度高的核心模块

重点审视以下高分支复杂度模块：

- `simulate/simulated_caching_allocator.py`
- `simulate/range_ops.py`
- `simulate/snapshot_mutator.py`
- `simulate/replay_executor.py`

识别方向包括：

- 重复错误处理模式
- 分支嵌套过深
- 单函数职责过多
- 关键逻辑难以单测驱动

### 6.3 覆盖率反向揭示问题

若某模块表现出“低覆盖 + 高复杂度”或“低覆盖 + 难测”，视为优先优化信号。此类优化以小步提炼为主，例如：

- 提取小型辅助函数
- 收敛重复校验逻辑
- 让失败路径更容易独立测试

## 7. 预期落地优化项

本轮预期会优先落地以下几类优化：

1. **测试启动方式标准化**
   - 移除 `ut/test.py` 这类基于路径注入的入口
   - 统一以 `pytest` 执行

2. **测试数据路径统一**
   - 所有测试通过稳定的文件路径解析测试数据

3. **工具入口适配标准包结构**
   - 确保模块方式运行不依赖仓库根目录

4. **核心模拟模块轻量降复杂度**
   - 收敛重复错误日志/返回模式
   - 必要时提取辅助函数以提升可读性与可测性

5. **关键低覆盖分支补测**
   - 尤其是错误路径、边界路径与容错路径

## 8. 实施顺序

### 阶段 1：建立标准可安装骨架

- 新建 `pyproject.toml`
- 建立 `src/memsnapdump/`
- 将源码迁入新包结构
- 修正基本 import

### 阶段 2：迁移并跑通测试

- `ut/ -> tests/`
- `test-data -> tests/test_data/`
- 接入 `pytest`
- 跑通现有测试

### 阶段 3：统计覆盖率并识别缺口

- 执行全量测试与 coverage
- 按模块识别低覆盖热点
- 确定补测优先级

### 阶段 4：小幅优化与补测联动推进

- 仅改高收益、低风险优化点
- 每项优化尽量由测试保护
- 补齐关键分支测试后回归验证

### 阶段 5：更新文档

- 更新 README 的安装、测试、运行说明
- 增加覆盖率执行方式说明

## 9. 错误处理策略

### 9.1 总体原则

- 尽量保持现有核心方法的成功/失败语义不变
- 不因迁移而大规模改写异常体系
- 优先增强路径、导入、入口类错误的可定位性

### 9.2 核心逻辑错误处理

对于 `simulate` 相关模块：

- 保持现有返回值语义（如布尔返回）优先
- 让失败路径更清晰、更稳定、更易测试
- 若出现重复错误模式，可做小范围收敛

### 9.3 工具入口错误处理

工具模块应保证：

- 文件不存在、参数非法、输出路径错误等情况能稳定报错
- 标准包结构下可运行
- 不依赖仓库根目录作为当前工作目录

## 10. 测试策略

### 10.1 迁移回归测试

验证目录迁移与框架切换未破坏现有能力：

- 关键模块可导入
- 原有测试迁移后可执行
- 主要工具入口可在新结构下运行

### 10.2 核心逻辑单元测试

重点保护：

- 成功路径
- 失败路径
- 边界路径
- 容错路径
- 状态转换路径

### 10.3 小规模集成测试

保留少量关键链路测试，例如：

- 小型 snapshot 回放
- hook 调用链路
- 工具处理最小输入样例

## 11. 验收标准

### 11.1 结构验收

- 源码迁移到 `src/memsnapdump/`
- 测试迁移到 `tests/`
- 项目支持 `pip install -e .`

### 11.2 测试验收

- `pytest` 可跑通测试集
- 覆盖率命令可执行并输出结果
- 至少补齐一批关键低覆盖模块测试
- 补测后全量测试仍通过

### 11.3 功能验收

- 关键模块 import 正常
- 主要工具入口在新结构下可运行
- 现有核心行为不因目录迁移发生明显变化

### 11.4 文档验收

- README 已更新安装、测试、运行方式
- 开发者可直接依据文档完成本地安装、测试与覆盖率执行

### 11.5 优化验收

- 完成一轮源码检视并明确记录优化点
- 实际落地优化以高收益低风险为原则
- 不引入额外测试错误或明显回归

## 12. 推荐实施方案结论

本次任务采用“标准化迁移 + 数据驱动补测 + 低风险优化”的组合方案：

- 先完成 `src-layout` 和标准安装能力
- 再完成测试体系切换和回归跑通
- 然后通过 coverage 结果指导补测
- 最后同步落地少量直接收益高的源码优化与文档更新

该方案兼顾了规范性、稳定性与实施成本，适合作为当前项目的一次基础设施升级。
